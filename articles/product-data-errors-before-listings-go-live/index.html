How to catch product data errors before listings go live

Product data quality checks should stop risky rows before they reach Shopify, Amazon Seller Central or eBay. The aim is not a perfect spreadsheet. It is a controlled release process.

Published 17 July 2026Updated 17 July 2026By Henry OliverUK ecommerce & wholesale
Quick answer

Product data quality checks before publishing ecommerce listings should test identifiers, commercial fields, required content and variant links. Each row should pass, warn or stop. A failed row needs a clear reason, owner and next action. Valid rows should continue, while uncertain or costly errors stay with a person.

Bad product data does not stay inside a spreadsheet. A reused EAN can attach work to the wrong product. A low price can damage margin. A missing size can break a variant family. A bad image link can leave a listing incomplete.

I ran a 200,000-SKU operation across Shopify, Amazon and eBay. At that scale, a final glance cannot protect the catalogue. You need rules that can be tested, explained and run again after a correction.

Which product data errors should stop a listing?

Start with the failures that create financial loss, channel rejection, customer confusion or hours of repair. Do not begin by checking every field with equal weight.

Identity

Check SKU, EAN or UPC, brand and manufacturer references. Duplicate or conflicting identifiers should stop the affected row.

Commercial data

Check selling price, cost, VAT code and margin. A price below cost or outside an approved tolerance needs review.

Required content

Check the title, images, category and required attributes for the target channel. Missing mandatory data should not publish.

Product relationships

Check parent and child links, size, colour, pack quantity and option values. One broken relationship can damage a whole variant group.

Rank the rules by consequence. A missing optional bullet point may be a warning. A duplicate identifier, negative price or missing parent SKU is a stop.

How do you turn a product rule into a check?

Name the mechanism. “Check the file” is not a specification. A developer and an operations manager need to know what the workflow will test.

  • Required-field check: confirm that the value exists and is not only spaces.
  • Allowed-value list: accept only agreed values such as active tax codes, sizes or channel categories.
  • Lookup table: match supplier brands, colours or product types to the approved internal value.
  • Duplicate check: compare the SKU or identifier against the file and the current catalogue.
  • Pattern check: confirm that identifiers, dates or URLs follow the expected structure.
  • Tolerance formula: compare price, cost or margin with an approved limit.

For each rule, record the field, test, severity, owner and repair action. That small control sheet becomes the build brief and the test plan.

Useful rule: if the team cannot explain why a row failed, the check is not ready for live use.

Why should each row pass, warn or stop?

A single fail result is too blunt. One bad row can hold up a whole launch, or the team may ignore the report so the upload can finish.

  1. Pass: all hard rules succeed. The row can move to the next stage.
  2. Warning: the row can move, but a person should review a non-critical issue.
  3. Stop: the row cannot publish because the error may block the channel, harm margin or mislead a customer.

Warnings need limits. If the team receives hundreds every day, they become background noise. Review warning volumes during the pilot and remove rules that do not change a decision.

What should a useful exception queue show?

“Invalid data” sends the work back to the person who prepared the file. A useful exception points to the problem and gives them a route to fix it.

Show the SKU, source file, failed field, value received, rule, severity, owner and next action. Also record whether the source data, internal mapping or channel output needs changing.

Keep the original row beside the corrected version. When the same error returns next month, you can decide whether to repair the supplier mapping rather than fix each file by hand.

How should you test product data checks?

Use a hard sample, not a clean demo. Include a duplicate SKU, reused identifier, price below cost, broken image link, missing parent, unusual pack size and an unrecognised attribute.

Run 50 to 100 mixed rows through the checks. Confirm that valid records pass, risky records stop and warnings stay visible. Correct the failed rows and replay them through the same process.

During the pilot, review every result. Separate bad source data, missing business rules and faulty mappings. Each problem has a different owner and fix.

How do you calculate the payback?

Measure the current failure loop. Count time spent checking files, reading upload reports, repairing listings and repeating the upload.

Suppose a team prepares 160 products each week. Manual checks take two and a half minutes per product, or about 6.7 hours. At a loaded staff cost of £29 an hour, that costs roughly £193 a week.

If automated checks reduce review to 50 seconds per product, the task falls to about 2.2 hours. The team gets back 4.5 hours a week, worth about £130. Across 48 working weeks, that is more than £6,000 of capacity.

If a build costs £3,600, the time-only payback is about 28 weeks. Use your own figures and do not count avoided errors unless you can measure them.

When should product data checks stay manual?

Do not automate a low-volume process with no repeatable rules. A small monthly upload may need a strong checklist, not a custom build.

Keep judgement manual when the source changes every time, category ownership is unclear or staff disagree on the correct value. The first job is to settle the definitions and owners.

A good first candidate has steady volume, repeated failure types and clear repair rules. That is where automation can remove checking without hiding risk.

What should the finished workflow do?

The workflow should collect source data, map fields, apply named checks and assign each row a result. It should create an exception queue, allow corrected rows to be replayed, publish only approved records and keep an audit trail.

FAQs

What product data should be checked before publishing?

Check identifiers, commercial fields, required content, image links, variants and channel-specific fields. Focus first on errors that can block a listing, damage margin or mislead a customer.

How do you automate product data quality checks?

Turn each business rule into a named test such as a required-field check, lookup table, duplicate check, allowed-value list or tolerance formula. Give each row a pass, warning or stop result.

Should one product data error stop the whole file?

Usually not. Stop the affected rows and let valid records continue, unless the failure shows that the whole file or mapping cannot be trusted.

When is product data checking not worth automating?

Keep the process manual when volume is low, source formats change constantly or the business has not agreed the rules. Fix ownership and definitions before commissioning a build.
